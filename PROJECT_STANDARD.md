# Project Standard

Every portfolio repository should prove one specific competence.

Do not create one large vague project that hides your learning. Create focused repositories with clear evidence.

## Minimum Portfolio-Grade Standard

A repository is portfolio-grade only if it includes:

1. Clear problem statement.
2. Defined scope.
3. Setup instructions that another person can follow.
4. Architecture explanation.
5. Test or test evidence.
6. CI/CD pipeline.
7. Security considerations.
8. Decision log.
9. Known limitations.
10. AI usage note.
11. Evidence of debugging, review, or hardening.

## Repository Questions

Each repository must answer one competence question.

Examples:

- `systems-foundation-labs`: Can I understand the operating system, network, shell, Git, logs, and debugging basics?
- `secure-api-template`: Can I build and explain a secure, testable backend API?
- `cicd-delivery-lab`: Can I convert a commit into a controlled build artifact?
- `shift-left-security-pipeline`: Can I integrate security checks early in development?
- `software-supply-chain-lab`: Can I reason about dependencies, artifacts, signatures, SBOM, and provenance?
- `cloud-secure-baseline`: Can I deploy a system to cloud with least privilege and basic observability?
- `observability-incident-lab`: Can I detect, analyze, and recover from system failure?
- `ai-native-devsecops-workflow`: Can I use AI to accelerate engineering without losing ownership?

## Rule

A project is not yours merely because it runs.

It becomes yours when you can explain it, test it, secure it, debug it, and improve it.
